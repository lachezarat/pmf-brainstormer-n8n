Architecture overview

One intake, multiple specialist stages, visible state all the way through.

The public API returns quickly, the long-running work happens asynchronously, and the status endpoint exposes the lifecycle without holding a user-facing request open.

Ingress

API workflows

POST /pmf-brainstorm

Validates the payload, creates a run row, returns `202`, then starts the orchestration path.

GET /pmf-brainstorm-status

Loads the current run state and final payload without recomputing work.

POST /pmf-brainstorm-review

Accepts operator review decisions for runs that have moved into manual review.

Error workflow

Maps an n8n execution failure back to the run record and marks it failed with an error summary.

State

Persistence and visibility

Postgres audit store

`pmf_runs` captures the lifecycle. `pmf_stage_runs` captures warnings, attempts, raw payloads, and parsed outputs.

Review gate

Low-score or incomplete runs move into `awaiting_review`, then resolve through an explicit operator decision path.

Reliability posture

Structured output is treated as a contract to enforce, not marketing copy to trust.

This is the part clients usually care about once a workflow leaves demo-land. The system is designed to make failure states legible and to keep operators from mistaking partial success for clean execution.

JSON schema reliability

Model-side constraints plus workflow-side validation

Each Gemini stage is prompted with a schema and then parsed again in code. Required keys, allowed enums, nested object structure, and arrays are checked before the result is treated as valid.

Retry discipline

Retries for malformed output and transient API failures

A stage can retry instead of collapsing the entire system on the first bad response, but every degraded outcome is still persisted as a warning, not hidden.

Silent-failure prevention

Explicit incomplete states and operator review gates

If a stage cannot produce a compliant payload after retries, it returns an explicit incomplete shape. Final synthesis can move the run into `awaiting_review` rather than bluffing a clean pass.

Execution failure path

Dedicated error workflow mapped by n8n execution ID

True execution failures route through a separate error workflow, which updates the run state and stores an error summary in the database.

Auditability

Postgres as the durable source of truth

The system stores status, warnings, raw model responses, parsed payloads, and final synthesis so debugging is not dependent on scrolling through n8n execution logs.

Handoff quality

Generated workflow files and documented topology

The repo is structured for export, backup, regeneration, and technical handoff instead of leaving the implementation trapped in a single shared n8n account.

Airtable control plane

A non-technical operator layer that sits beside the workflow runtime instead of inside it.

For clients who need human review, ownership, prioritization, and a clean operational surface, I pair the Postgres audit store with an Airtable model. Postgres keeps the execution truth; Airtable becomes the control plane people actually work from.

01

Runs

Status, verdict, score, warnings, operator owner, and lifecycle timestamps.

02

Stage Attempts

Per-stage latency, warning counts, attempt number, and parsed-output references for debugging.

03

Gate Decisions

Human review state for degraded runs, approval notes, and intervention history.

04

Experiments

Validation-plan actions linked back to each run so execution does not stop at analysis.

05

Prompt Configs

Runtime-active prompt, schema, fallback, model, and retry settings loaded before each Gemini stage.

Why this matters

n8n is the orchestration engine, not the client’s day-to-day workspace. Airtable fills the gap when a delivery needs review queues, linked records, ownership, and light-weight reporting.

Recommended relationship

Keep Postgres as the durable audit trail. Use Airtable for runtime prompt config, operator workflows, approvals, experiment tracking, and handoff.

Design stance

Airtable should reflect system state, not become the only place where critical execution truth lives.

Deployment shape

Static front-end where it belongs. Long-running workflow runtime where it belongs.

Netlify is a good home for the portfolio layer. The n8n runtime, Postgres database, and webhook endpoints should live on a VPS or container host with persistent storage, managed secrets, and proper TLS.

Public surface

Netlify

Hosts this static case study, assets, and optional demo links. No workflow runtime is forced into a static platform.

Execution layer

VPS or container host

Runs `n8n + Postgres` via Docker Compose, exposes the async webhook API, and persists workflow state.

Operator layer

Airtable

Linked records for runs, stage attempts, experiments, and review-gate decisions.

Providers

Gemini and Exa

Research and structured AI stages stay behind the backend boundary so secrets are never embedded in the portfolio site.

Practical rule: deploy the site on Netlify, deploy the automation runtime somewhere stateful, and wire the two together with a public-safe demo URL or a contact-driven walkthrough.
